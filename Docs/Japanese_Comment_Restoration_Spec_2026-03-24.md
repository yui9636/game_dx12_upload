# 日本語コメント復旧仕様書

日付: 2026-03-24
対象: `C:\Users\yuito\Documents\MyEngine_Workspace\game_dx12_upload`

## 1. 目的

ソース内で文字化けしている日本語コメントを、Visual Studio のエンコード設定やプロジェクト設定を変更せずに、日本語コメントそのものを書き直して復旧する。

今回の目的はあくまで**可読性の復旧**であり、ロジック変更や API 契約変更は含めない。

## 2. 制約

- Visual Studio 側の文字コード設定は変更しない
- `.vcxproj` の文字コード関連設定は変更しない
- 自動変換ツールで一括再エンコードしない
- 文字化けコメントは**人間が読める日本語コメントとして手で置換**する
- コードの意味が不明な箇所は、無理に復元せず、簡潔で正しい新コメントに差し替える
- 変数名・関数名・識別子は変更しない
- 動作変更は行わない

## 3. 完了条件

以下を満たしたら完了とする。

1. `Source` 配下の主要 C++ ファイルに残る文字化けコメントを除去済み
2. 必要なコメントは日本語で自然に読める
3. 不要な冗長コメントは削除済み
4. ビルドが通る
5. コメント修正だけで挙動が変わっていない

## 4. 優先順位

### Phase 1: 主要実装ファイル

最初に、読む頻度と影響の大きい中核ファイルから直す。

- `Source\Actor\*`

- `Source` 配下の残り
- 必要であれば `Shader` 配下のコメントも対象にする

## 5. コメント復旧ルール

### 5.1 基本方針

- 元コメントを無理に完全復元しない
- 現在のコードを読んで、**今の実装に合った日本語**へ置き換える
- 「何をしているか」「なぜ必要か」が伝わる短いコメントにする

### 5.2 残すコメント

- 意図がコードだけでは読み取りにくい箇所
- DX11 / DX12 の差分理由
- ライフタイムや順序依存
- バリア、フェンス、descriptor など事故りやすい箇所

### 5.3 削るコメント

- コードを読めば明らかな逐語説明
- 古い仕様に基づく誤誘導コメント
- 実装と食い違うコメント
- 文字化けしていて意味が壊れているコメント

### 5.4 コメント文体

- 日本語で簡潔に書く
- 断定的に短く書く
- 句読点は最小限
- 1 行または短いブロックを基本とする

例:

- 悪い例: `// ここで変数xに値を代入する`
- 良い例: `// 前フレームの状態を保持して補間に使う`

## 6. 文字コード方針

今回の対処は**再エンコードではなく書き換え**とする。

つまり:

- 文字化けしたコメントを検出
- そのコメント行を削除または置換
- 新しい日本語コメントを書き込む

これにより、IDE 設定やビルド設定に依存せず復旧する。

## 7. 実施手順

1. 文字化けコメントを検索
2. ファイルごとに意味のあるコメントだけを残す
3. 必要な箇所だけ日本語で再記述
4. 差分を確認
5. ビルド
6. 問題なければ次のフォルダへ進む

## 8. 検出方法

文字化けコメントの代表例:

- `���`
- `�e�`
- `縺`
- `繧`
- `繝`
- 文字化けした日本語っぽい記号列

ただし、検索結果のすべてを機械的に修正しない。
`#include` 行などで文字コード由来の誤検出が混ざる場合があるため、必ず中身を見て判断する。

## 9. リスク

### 9.1 コメントだけのつもりでロジックを壊す

対策:

- コメント以外は触らない
- インデント整理やフォーマット変更も最小限にする

### 9.2 元の意図を誤って再記述する

対策:

- 元コメントの復元ではなく、現在コードの実挙動ベースで書き直す
- 意味が曖昧な箇所は無理に長文を書かない

### 9.3 文字コード混在をさらに悪化させる

対策:

- 一括変換しない
- 行単位・ブロック単位で差し替える

## 10. 対象外

今回の仕様では以下は行わない。

- Visual Studio の設定変更
- `.editorconfig` やプロジェクト全体の文字コード統一
- 識別子の日本語化
- ドキュメント全体の全面改稿
- ログ文字列や UI 文言の全面見直し
